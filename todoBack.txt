Общая концепция серверной части
Сервер будет отвечать за:
Регистрацию, авторизацию и подтверждение электронной почты.
Хранение данных о пользователях, их уровне, инвентаре, валюте, прогрессе.
Запуск логики матчмейкинга (поиск игроков для PvP, генерация инстансов для PvE/PvP).
Хранение результатов боёв, начисление наград, прогрессии.
База данных (SQL или NoSQL) для постоянного хранения состояния (пользователи, предметы, матчмейкинг-очередь, история боёв).
Технологии (примерный выбор)
Backend: Node.js + Express или Next.js API Routes.
БД: PostgreSQL (через Prisma) или MongoDB (через Mongoose).
Аутентификация: JWT или session-based auth + отправка email через внешние сервисы (SendGrid, Mailgun).
Деплой: Vercel (для Next.js фронтенда) + бесплатные варианты для Node.js backend (например, Render или Railway) + бесплатный Managed DB (Neon.tech для Postgres).
Поэтапный план
Этап 1: Подготовка окружения для разработки сервера
Создание репозитория

Отдельный репо для backend или внутри того же монорепо, но в отдельной папке server/.
Инициализация Node.js проекта.
Настройка TypeScript для сервера

Добавление tsconfig.json, установка typescript, @types/node.
Выбор фреймворка

Например, Express.js или Next.js API Routes. Если вы уже используете Next.js, можно использовать встроенные API routes для упрощения деплоя (меньше конфигурации).
Этап 2: Настройка базы данных
Выбор БД

Пусть это будет PostgreSQL.
Подключение к БД

Используйте Prisma или Knex для миграций и запросов к БД.
Создайте таблицы:
users (id, email, hashed_password, is_verified, level, currency, …)
inventory_items (user_id, item_type, count, …)
instances (id, mode, start_time, end_time, …)
players_instances (instance_id, user_id, position, stats, …)
matchmaking_queue (id, user_id, mode, level, ...)
Миграции

Создайте миграции для создания таблиц.
npx prisma migrate dev (если Prisma).
Локальное тестирование

Поднимите локально Postgres через Docker или используйте локальную установку.
Проверка подключения и базовых запросов.
Этап 3: Аутентификация и регистрация пользователей
Регистрация пользователя

POST /api/auth/register (email, password)
Хэширование пароля (bcrypt).
Создать запись в БД users с is_verified = false.
Отправка email с подтверждением

Генерация одноразового токена для верификации email.
Использование внешнего сервиса (SendGrid) для отправки письма с ссылкой типа /api/auth/verify?token=....
Верификация email

GET /api/auth/verify?token=...
Проверить токен в БД, отметить is_verified = true.
После верификации перенаправить на страницу входа.
Авторизация (Логин)

POST /api/auth/login (email, password)
Проверить пароль, если ок — выдать JWT или установить сессию.
Возвращать токен клиенту, хранить в localStorage или cookie httpOnly.
Middleware для защищённых маршрутов

Любые игровые запросы (матчмейкинг, запуск инстанса) требуют авторизации.
Этап 4: Связь клиент ↔ сервер
Интеграция фронтенда с auth API

На ModeSelectionPage после выбора режима и клика "Начать подбор" отправить запрос на сервер:
Если не авторизован — перенаправить на страницу логина.
Matchmaking

POST /api/matchmaking/join (параметры: mode)
Сервер ставит игрока в очередь, возвращает instanceId когда готово.
Клиент после получения instanceId переходит на /game?instanceId=....
Загрузка инстанса

GET /api/instance/{instanceId}
Возвращает режим, список игроков и их начальные параметры, карту (или seed для генерации карты на клиенте), или саму сгенерированную карту.
Отправка игровых действий

Когда игрок двигается или атакует:
Отправлять POST /api/instance/{instanceId}/action (payload: действие, направления, etc.)
Сервер обновляет состояние инстанса (в БД или в памяти), возвращает обновлённое состояние.
Обновление состояния

При каждом действии клиент запрашивает новое состояние через /api/instance/{instanceId}/state или использует WebSocket для подписки на обновления (в реальном PvP лучше WebSocket).
Этап 5: Добавление WebSocket для PvP
Настройка WebSocket сервера

Использовать например Socket.IO.
При подключении по instanceId игрок получает обновления в реальном времени о действиях других игроков.
Клиентский код

Подключение к WebSocket при входе в инстанс.
При получении событий "player_moved", "artifact_picked", "battle_result" обновлять локальный стейт.
Этап 6: Артефакт, бочки, портал
Логика на сервере

Когда игрок открывает бочку (/action {type:"open_barrel", direction:{dx,dy}}):

Сервер решает результат: артефакт, монстр или ресурс.
Обновляет состояние игрока (добавляет в инвентарь), или порождает бой с монстром.
Артефакт в БД:

Храните в instances поле artifactOwner. Если null — артефакт ещё не получен.
Когда артефакт подбирают, artifactOwner = player_id, оповестить всех участников через WebSocket.
Портал:

Если игрок стоит на портале и у него есть артефакт, по действию exit_portal сервер завершает инстанс, начисляет награды (progressionSystem).
progressionSystem.ts

На сервере: функция finalizeInstance(instanceId) подсчитывает результаты, обновляет users таблицу, добавляет опыт и валюту.
Возвращает результат всем игрокам.
Этап 7: Прогрессия и улучшения
Покупка предметов

POST /api/shop/buy_item (авторизованный запрос)
Проверяет валюту игрока в БД, уменьшает её, добавляет предмет в инвентарь.
Улучшение характеристик

POST /api/player/upgrade_stat (например, повысить maxEnergy)
Сервер обновляет запись игрока.
Этап 8: Тестирование локально
Регистрация, подтверждение email

Тестировать с временными email сервисами или локально отключить отправку писем, а только логировать ссылку.
Авторизация

Проверить вход, выход, хранение токена.
Матчмейкинг

Подключить нескольких тестовых игроков, проверить выдачу instanceId.
Инстанс

Проверить генерацию карты, взаимодействия с бочками, монстрами, артефактом.
Проверить выход через портал, завершение инстанса, начисление наград.
Этап 9: Деплой
Backend (API) деплой

Использовать бесплатный хостинг, например Render или Railway.
Задеплойте Node.js приложение с командой npm run start.
Подключите бесплатную Postgres БД (Neon.tech).
Frontend (Next.js)

Задеплойте на Vercel.
Настройте переменные окружения для API URL, ключей для почтового сервиса.
CORS и безопасность

Настройте CORS, чтобы фронтенд с Vercel мог обращаться к API на Render.
Включите HTTPS.
Проверка на продакшне

Зайдите на URL, зарегистрируйтесь, получите письмо (на реальную почту).
Проверьте матчмейкинг (пригласите друга).
Этап 10: Тестирование с друзьями
Отправьте ссылку на Vercel деплой.
Друзья регистрируются, выбирают режим (1v1) и попадают в один инстанс.
Проверить бой, сбор артефакта, выход через портал.
По завершению инстанса проверить начисление опыта и валюты.
